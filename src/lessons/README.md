# Lesson Authoring Guide

This guide covers how to create and validate lessons for the SPE interactive tutorial.

## Task YAML Anatomy

Each lesson is a YAML file with metadata and a `tasks` array. Each task has:

```yaml
- instruction: |          # What the user should do (markdown, required)
  hint: |                 # Full answer shown after 4 failed attempts (required)
  nudge: |                # Gentle hint shown after 2 failed attempts (optional)
  starterCode: |          # Pre-filled editor content for ise/builder mode (optional)
  validation:             # How completion is determined (required)
  successMessage: |       # Explanation shown on success (optional)
  builderConfig:          # Visual builder config, builder mode only (optional)
```

## Three Validation Types

### `structural` ‚Äî No execution, checks command shape

Checks cmdlet name, resolved path (by node ID), and switches. Best for single-cmdlet tasks teaching navigation.

```yaml
validation:
  type: structural
  cmdlet: get-item
  requirePath:
    - "master:\\content\\Home"
    - "master:/content/Home"
    - "master:\\Content\\Home"
  requireSwitches:
    - recurse
  requireParams:
    Name: "Contact Us"
    ItemType: "Sample/Sample Item"
```

**Fields:**
- `cmdlet` (required) ‚Äî lowercase cmdlet name (e.g. `get-childitem`)
- `requirePath` (optional) ‚Äî array of path variants; resolved to node IDs for comparison
- `requireSwitches` (optional) ‚Äî array of switch names without the dash prefix
- `requireParams` (optional) ‚Äî map of parameter name ‚Üí expected value (case-insensitive key lookup, exact value match)

**Path arrays should include multiple format variants** (backslash, forward-slash, lowercase) since the validator resolves them to node IDs.

**Error cascade:** wrong cmdlet ‚Üí invalid path ‚Üí wrong path ‚Üí missing switch ‚Üí missing/wrong param

### `pipeline` ‚Äî Checks stages + optionally executes for output

Checks that required cmdlets appear anywhere in the pipeline (alias-aware), then optionally runs the script to verify output.

```yaml
validation:
  type: pipeline
  stages:
    - get-childitem
    - where-object
    - select-object
  outputContains: "Sample Item"
  outputNotContains: "Folder"
```

```yaml
# With parameter checking across multi-command scripts
validation:
  type: pipeline
  stages:
    - new-item
    - get-childitem
  requireParams:
    Name: "Contact Us"
    ItemType: "Sample/Sample Item"
  outputContains: "Contact Us"
```

**Fields:**
- `stages` (required) ‚Äî array of lowercase cmdlet names checked across ALL statements
- `requireParams` (optional) ‚Äî map of parameter name ‚Üí expected value; matched against ANY stage across all statements (case-insensitive key lookup, exact value match)
- `outputContains` (optional) ‚Äî string that must appear in execution output
- `outputNotContains` (optional) ‚Äî string that must NOT appear in execution output

**Aliases are accepted:** `gci` matches `get-childitem`, `?` matches `where-object`, etc.

**Error cascade:** missing stage ‚Üí missing/wrong param ‚Üí wrong output ‚Üí filter too broad

### `output` ‚Äî Execute and check output only, no structural checks

Runs the script and checks output strings. Best for run-and-observe tasks where the starterCode is complete.

```yaml
validation:
  type: output
  outputContains: "Report"
  outputNotContains: "Error"
```

**Fields:**
- `outputContains` (optional) ‚Äî string that must appear in execution output
- `outputNotContains` (optional) ‚Äî string that must NOT appear in execution output

**Use for:** Tasks with complete `starterCode` where the user runs pre-written code to observe behavior.

**Error cascade:** output mismatch

## Three Editor Modes

Set via the lesson-level `mode` field:

- **`repl`** (default) ‚Äî Single-line console. No starterCode. User types commands from scratch.
- **`ise`** ‚Äî Multi-line editor. StarterCode can be partial (user completes it) or complete (user runs and observes).
- **`builder`** ‚Äî Visual drag-and-drop pipeline builder. Uses `builderConfig` for available cmdlets and locked prefilled stages.

## Error Messages Generated by Validator

| Scenario | Message |
|---|---|
| Wrong cmdlet | "Try using `Get-Item` for this task." |
| Invalid path | "The path you specified doesn't resolve to a valid item. Check your path." |
| Wrong path | "Your command resolved to the wrong item. Check your path." |
| Missing switch | "Don't forget the `-Recurse` switch." |
| Missing param (structural) | "Don't forget the `-Name` parameter." |
| Wrong param value (structural) | "The `-Name` parameter value doesn't match what's expected." |
| Missing pipeline stage | "Your script should include `Where-Object`." |
| Missing param (pipeline) | "Your script should include `-Name "Contact Us"`." |
| Wrong output (pipeline) | "Your script runs, but the output doesn't match what's expected. Check your filter criteria." |
| Filter too broad | "Your filter isn't specific enough ‚Äî unwanted items are getting through." |
| Wrong output (output) | "Your script runs, but the output doesn't match what's expected. Check your code." |

## Hint Tier System

| User Attempts | UI Behavior |
|---|---|
| 0‚Äì1 | No hints available |
| 2‚Äì3 | "Show Hint" button ‚Üí displays `nudge` (gentle guidance) |
| 4+ | "Show Answer" button ‚Üí displays `hint` (full solution as code block) |

## Validation Design Principles

1. **Accept equivalent input** ‚Äî paths compared by node ID, cmdlets accept aliases, extra params ignored
2. **Progressive feedback** ‚Äî `partial[]` shows checkmarks for correct parts
3. **Guide without revealing** ‚Äî error messages point to the problem area, never show the answer
4. **Validate intent, not syntax** ‚Äî "did they use the right tool on the right target?"

## Lesson Authoring Checklist

- [ ] Every task has `instruction`, `hint`, and `validation`
- [ ] Path arrays include multiple format variants (backslash, forward-slash, lowercase)
- [ ] `outputContains` uses a string unique to the correct output
- [ ] `outputNotContains` catches the most common wrong answer
- [ ] `successMessage` explains the "why" or teaches a concept
- [ ] ISE tasks have `starterCode` (partial for learning, complete for observation)
- [ ] Builder tasks have `builderConfig` with `availableCmdlets` and any `prefilled` locked stages
- [ ] `nudge` provides directional guidance without giving the answer
- [ ] `hint` provides the complete working command/script
